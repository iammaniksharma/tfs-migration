Part 1 — Survival Checklist (8 Rules)

These are not suggestions. Skip any of these and your migration will fail.

RULE 1
Every flow must live inside a named Solution — never the Default Solution.
  • Go to Solutions in the CRG tenant left nav.
  • If your flow is under My Flows or Default Solution, move it before doing anything else.
  • Create a new solution: Solutions → New Solution.
  • Name it clearly. Example: CRG_MIG_[YourInitials]_[Date]
  • Add the flow: Add Existing → Automation → Cloud Flow.
  • Confirm the flow is inside your named solution before proceeding.
RULE 2
Use Connection References — not direct connections inside flows.
  • Open each flow inside your solution.
  • Check every connector action (SharePoint, Outlook, Teams, SQL, etc.).
  • If you see a direct named-user connection, replace it with a Connection Reference before export.
  • Add a reference: Solutions → [Your Solution] → Add New → More → Connection Reference.
  • Edit the flow. Replace the direct connector with the connection reference.
  • Direct connections will break on import to TFS. Do not skip this.
RULE 3
Declare all Environment Variables inside the solution before export.
  • Check your flows for hardcoded values: URLs, site addresses, list names, IDs, email addresses, tenant-specific strings.
  • For each hardcoded value, create an Environment Variable: Solutions → [Your Solution] → Add New → More → Environment Variable.
  • Replace the hardcoded value in the flow with the environment variable.
  • Record all variable names and their expected TFS values in the mapping table (see Part 3a).
  • At minimum, replace: SharePoint site URLs, tenant domain names, email addresses, and CRG-specific IDs.
RULE 4
Export as Unmanaged from CRG.
  • Never export as Managed from the source tenant. Managed solutions are locked — you cannot edit them after import.
  • Go to Solutions → [Your Solution] → Export Solution.
  • Select Unmanaged.
  • Download the .zip file. Name it: CRG_MIG_[FlowName]_Unmanaged_[Date].zip.
  • Keep this file. Do not modify the zip.
RULE 5
Check for Child Flows before you export.
  • Open each flow. Look for the action Run a Child Flow anywhere inside it.
  • Every child flow referenced must also be inside the same solution.
  • Add missing child flows: Solutions → [Your Solution] → Add Existing → Automation → Cloud Flow.
  • Export fails silently on missing child flows. The parent imports but throws a runtime error.
RULE 6
Import into TFS as Unmanaged.
  • Log into Power Automate in the TFS tenant.
  • Go to the correct environment (Dev / Test / Prod). Do not import into the wrong one.
  • Navigate to Solutions → Import Solution. Upload your .zip file.
  • Select Unmanaged on import.
  • On the Connection References screen: map every reference to a valid TFS connection. Create new connections if they don't exist. Use service accounts where required.
  • On the Environment Variables screen: enter the correct TFS values. Do not leave any variable blank.
RULE 7
Activate flows manually after import.
  • After import completes, go into your solution in TFS.
  • Open each flow. Most flows import as Off.
  • Click Turn On.
  • Run a manual test immediately after turning on.
RULE 8
Validate before calling it done.
  • Trigger each flow manually at least once.
  • Check the Run History for a successful run.
  • Confirm all connections are authenticated (no 401 or credential errors).
  • Confirm all environment variables resolved correctly.
  • If a flow uses a desktop machine, confirm the machine is registered and online in TFS.
  • Sign off only after a clean successful run.

Pre-Export Checklist

  • Flow is inside a named solution (not Default Solution, not My Flows)
  • All child flows are inside the same solution
  • All connectors use Connection References (no direct connections)
  • All tenant-specific values use Environment Variables
  • Solution name is clearly labeled for identification
  • You know which TFS environment to target (Dev / Test / Prod)
  • You have access to the TFS environment and can create connections

Post-Import Validation Checklist

  • Solution is visible in TFS environment
  • All flows are present in the solution
  • All Connection References are mapped and authenticated
  • All Environment Variables have TFS values entered
  • Each flow has been turned on manually
  • At least one successful manual run recorded in Run History
  • No 401 errors, no missing connection errors, no null variable errors
  • If RPA flow: desktop machine or machine group is assigned and online

Part 2 — Explicit Approval to Ignore Best Practices (For Now)

This is a migration, not a rewrite. You have permission to migrate messy, non-standard, or poorly structured flows. Your only goal is to make them run in TFS.

✓ SAFE TO IGNORE

  • Bad naming conventions (flow names, variables, actions)
  • Missing error handling or try/catch scopes
  • No comments or descriptions in flows
  • Inconsistent structure, spaghetti flows, deep nesting
  • Hardcoded values that are not tenant-specific
  • Missing logging or run tracking
  • Non-standard trigger conditions or schedules
  • Performance inefficiencies (loops in loops, redundant calls)
  • No documentation or readme

✗ WILL BREAK THE FLOW

  • Direct connections (must use Connection References)
  • Missing child flows in solution
  • Hardcoded CRG tenant URLs, domain names, email addresses, IDs
  • Flow sitting in Default Solution or My Flows
  • Desktop flows with no machine assigned in TFS
  • Flows depending on connectors not available in TFS
  • Hardcoded credentials in HTTP actions or connection strings — flag to lead immediately
The Rule: Move First. Fix Later.

If it runs, you are done for now. Every other issue is a post-migration backlog item. Do not let perfectionism block delivery.

Code Review Checklist

  • Flow names follow agreed naming convention
  • All actions have meaningful names (not "Condition 2", "Apply to each 3")
  • Error handling scope added around critical actions
  • Sensitive data not logged in run history
  • No redundant or duplicate steps
  • Parallel branches used where appropriate
  • Recurrence triggers have correct time zones set

Post-Migration Improvements (Backlog)

  • Replace Apply to each loops with Select and Filter Array where possible.
  • Use batch requests for SharePoint or Dataverse operations at high volume.
  • Add error handling scopes with Teams or email notifications on failure.
  • Add filter queries to triggers to reduce unnecessary runs.
  • Add concurrency controls on loops to prevent throttling.

Part 3c — Testing Strategy

Environments and Machine Types

Development
HM — Hosted Machine

Microsoft-provided HM in Power Automate Portal. Initial testing and post-import validation.

Test
HMG — Machine Group

Regression and integration testing. Unattended execution.

Validation (UAT)
HMG — Machine Group

User acceptance testing. Sign-off required before Production.

Production
HMG — Machine Group

Live operations. At least one machine online at all times.

Testing in Development Using HMs

  • Every flow runs at least once with a successful result.
  • All Connection References authenticate without errors.
  • All Environment Variables resolve to the correct TFS values.
  • Child flows are reachable and execute correctly.
  • Desktop flows: the HM is assigned and the flow completes end-to-end.
  • To test on HMs: go to Monitor → Desktop flow runs. Trigger the flow manually. Confirm the HM picks up the run (status: Running). Check result: Succeeded or Failed.
  • For attended flows on HMs: the session must be active and unlocked.

Minimum Dev Test Cases

  1. Happy path — trigger with valid input, confirm expected output.
  2. Null/empty input — confirm flow handles gracefully (or note as a known gap).
  3. Connection test — confirm all connections authenticate on first real run.

Testing in Test and Production Using HMGs

  • Machine group must be assigned to the desktop flow in TFS.
  • At least one machine in the group must show Online status.
  • The service account used must have permissions on the target application.
  • The flow must complete unattended — no prompts, pop-ups, or auth dialogs during execution.

Minimum Test/Prod Validation Cases

  1. Flow triggers correctly from its source (schedule, button, upstream event).
  2. The HMG picks up the desktop run within expected wait time.
  3. End-to-end data accuracy confirmed — input matches expected output.
  4. Run during off-peak hours first to confirm machine availability.

Failure Handling and Rollback

If a flow fails after migration:

ErrorFix
Invalid connection / 401 UnauthorizedRe-authenticate the Connection Reference
Environment variable is empty or nullSet the variable value in the TFS solution
Child flow not foundAdd child flow to solution, re-import
Machine unavailable (RPA)Check machine/group status, assign correct machine
Permission denied on target systemUpdate service account permissions
Flow is OffTurn on the flow manually
Cannot fix quicklyTurn off in TFS, notify lead, re-run from CRG as fallback

Rollback steps:

  • Keep CRG flows active until TFS is fully validated and signed off.
  • Do not deactivate CRG flows until sign-off is confirmed in TFS.
  • If TFS fails and CRG needs to resume: re-enable the CRG flow immediately. No re-import needed.
  • Document the rollback trigger condition and who has authority to initiate it.
